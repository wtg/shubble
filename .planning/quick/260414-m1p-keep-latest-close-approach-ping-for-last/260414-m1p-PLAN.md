<objective>
Keep the LATEST close-approach ping (timestamp max) per (vehicle_id, stop_name) in `_compute_vehicle_etas_and_arrivals`, not the globally-closest one (distance min). The current closest-approach dedup discards the current loop's detection whenever an earlier loop happened to hit the stop geometrically closer, causing `last_arrivals_by_vehicle[vid][stop]` to hold a stale timestamp that then fails the `loop_cutoff` filter in `build_trip_etas`, leaving the current trip's stop with `last_arrival=None`. Houston Field House never flips to "Last:" because every loop's current-loop detection loses the dedup to an earlier loop's closer ping.

Purpose: Fix the "HFH never gets Last:" persistent bug and two related symptoms (WEST stops flipping back to live-ETA after being passed; multi-loop detections never making it through the UI pipeline). The 60m CLOSE_APPROACH_M filter still ensures every surviving row is a genuine arrival, so "latest" vs "closest" among already-valid rows is just a tiebreaker choice — "latest" is the only choice that survives the downstream loop-scoping filter.

Output: Two-line sort-key change in `backend/worker/data.py`, renamed local variable, rewritten rationale comment block, and one new regression test file.
</objective>

Root cause (from the task_context — do NOT re-investigate):

- `backend/worker/data.py:1192-1200` sorts `close_rows` by `_dist_m` ascending and keeps `first`. Across a full day of polling, this keeps whichever loop's ping happened to land geometrically closest to the stop — typically an earlier loop, not the current one.
- Downstream, `backend/worker/trips.py:215-226` applies a `loop_cutoff` filter (= `actual_departure` for active trips, `prior_departure` for completed trips) that DROPS any `last_arrival` older than the cutoff as a prior-loop leak.
- Net: the current loop's detection is discarded by the dedup, the earlier loop's detection is then discarded by the `loop_cutoff` filter, and the trip's `stop_etas[stop].last_arrival` is `None` — even though the shuttle passed the stop moments ago.

Evidence from live env (vid=001, NORTH, HOUSTON_FIELD_HOUSE):
| time | dist to HFH |
|------|-------------|
| 18:15:04 | 2m ← wins the current closest-distance dedup |
| 19:48:08 | 18m ← current loop, silently discarded |

After the fix, the 19:48:08 timestamp wins (latest), passes `loop_cutoff=actual_departure`, and HFH shows Last: on the UI.

Why "latest" is correct as a tiebreaker:
- Every surviving row has already passed the 60m `CLOSE_APPROACH_M` filter — they are all genuine arrivals. The earlier "closest-approach" rationale was about rejecting far-tagged polyline artifacts; that work is now done entirely by the 60m filter.
- The latest within-60m ping belongs to the most recent loop by definition, so it naturally passes `loop_cutoff` for the current active trip.
- For completed trips (which use `prior_departure` as `loop_cutoff`), the latest-within-60m ping that survives that cutoff is still the tip of the prior loop — correct behavior.
- Older detections are no longer useful — the UI only cares about the current loop's arrival per trip.

Why "latest" not "closest-distance":
- Every close_rows row is already verified within 60m of the stop's real coord by CLOSE_APPROACH_M. Distance tiebreaking between two already-valid pings has no semantic value — it just picks an arbitrary loop's ping.
- Timestamp tiebreaking has a meaning that matches the downstream contract: `build_trip_etas` wants the detection from THIS loop, which is the latest one that fits within the loop's time window.
- `_dist_m` column is now unused for the dedup but is fine to keep (negligible cost, leaves room for future debugging).

Why NOT other approaches:
- Could pre-filter to "only pings within the current loop window" here — but the data layer doesn't know per-trip windows; that's `build_trip_etas`' job. Stay with the existing separation of concerns.
